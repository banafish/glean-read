# 技术设计：GleanRead 系统

## 上下文 (Context)
当前开发者或技术人员在移动端的碎片化阅读内容难以沉淀，流失严重。为解决这个问题，本系统名为 **GleanRead**，聚焦于以最小阻力在手机端收集“知识碎片与个人想法”，并在 PC 端结合 AI 能力将高频标签下的碎片总结合并，挂载至树形知识网络中。

## 目标 / 非目标 (Goals / Non-Goals)
**目标:**
- 支持移动端 Android 系统级的快捷分享拦截，降低分享心智负担。
- 基于 Spring Boot + PostgreSQL（含 pgvector）构建灵活的后端架构。
- 提供大屏（PC Web）视角的标签云及树状节点管理。
- 引入大语言模型（如 DeepSeek/通义）实现对零散摘抄的“结构化提纲合并”。

**非目标:**
- 全文检索的搜索引擎（重心在于个人化的高亮碎片，不爬取存储数万字全文）。
- 离线优先的重型客户端（初版强依赖网络与云端 Inbox 同步）。

## 决策 (Decisions)
1. **客户端接入**: Android 应用使用 Jetpack Compose。利用 `Intent.ACTION_SEND` 捕获分享，调起透明 Activity/Service 显示半屏浮层，实现即用即走的“快门式”采集。
2. **后端与持久化**: 使用 Kotlin/Java + Spring Boot 3。数据库选择 PostgreSQL（JSONB 可储存灵活的数据上下文结构，支持未来扩展向量查询）。
3. **AI 合成方案**: 不由移动端本地模型运行。由 PC 端或后端发起的 API 调用，将同一个标签的 M 个碎片发送给 LLM 执行 Prompts，要求 LLM 输出包含特定标识的大纲。
4. **异步队列**: Redis 作为高频打标统计、系统级防抖和 Inbox 的临时缓冲层。

## 风险 / 权衡 (Risks / Trade-offs)
- **风险**: 移动端厂商对后台悬浮窗/弹窗保活限制严格。
  - **缓解**: 优先使用系统原生分享面板，弹出独立的、仅占半屏的透明 Activity，确保可靠性。
- **风险**: 碎片过于混乱，AI 梳理时极易产生幻觉。
  - **缓解**: Prompt 设计时，强制要求 AI 仅基于提供的碎片 Context 提取，并以特定的编号格式指向原文来源。
